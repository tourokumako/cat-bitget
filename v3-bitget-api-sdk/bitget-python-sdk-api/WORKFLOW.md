## 作業フロー（必ず順番通りに実行）

```
Step 0.  継承範囲固定    — V9の「残すもの / 変えるもの」を明示してからスタート
                          毎サイクル開始時に確認する

Step 1.  構造分解診断    — 以下の4指標を分離して測る
                          ① 件数不足か
                          ② 保有時間が長すぎて機会損失か
                          ③ fee込みで期待値がマイナスか
                          ④ Entry品質（偽シグナル率）が悪いか
                          → 「何がボトルネックか」を1つに絞る

Step 2.  1テーマ仮説設計 — Exit / Entry どちらか1テーマのみ設計する
                          （Exit → Entry の順。同時変更禁止）
                          以下の事前検証ゲートを全て通過しない限り提案しない:
                          □ 変更後の件数が ≥ 20件/day になるか（既存CSVで試算）
                          □ Bitget本番で観測・執行可能な条件か
                          □ fee込みで期待値がプラスになるか
                          □ LONG/SHORT どちらを壊すか事前に点検済みか

Step 3.  ユーザー承認    — 以下の5点をセットで提示してから止まる  ← STOP
                          ① 目的（何を改善するか）
                          ② 根拠データ（数値ベースの分析結果）
                          ③ 変更対象（Before / After）
                          ④ 予想される改善（件数・NET・TP率の試算）
                          ⑤ 壊れる可能性がある箇所

Step 4.  最小差分実装    — 1変更のみ。コード差分を提示してGO待ち

Step 5.  Replay実行      — 90日データで実行  ← STOP

Step 6.  採用判定        — 全項目チェック後「採用しますか？」← STOP

Step 7.  lessons 更新    — 想定外の結果・禁止パターン発見時のみ

Step 8.  baseline 更新   — 採用時のみ WORKFLOW.md を更新

Step 9.  Git コミット    — ユーザーの明示的な OK が出てからコミット  ← STOP
```

> **STOP のルール**: 各 ← STOP では必ずユーザーに結果を提示し、
> 明示的な承認（"GO" / "OK" / "採用" 等）を得てから次に進む。
> 曖昧な返答・質問への回答は GO ではない。

```

  0.024 BTC → $0.64/TP    0.12 BTC → $3.19/TP ← 上限フル活用

件数別 NET/day 上限（損失ゼロの理想値）:
              9件/day   15件/day   20件/day
0.024 BTC     $5        $9         $12
0.12 BTC      $27       $45        $61 ← $60/day達成ライン

最短ルート: 0.12 BTC × add=1 × 20件/day = $61/day
ただし損失もスケール → 品質改善が先決
```
