### L-135: HMM の安定性(ARI)と分離度はトレードオフ（2026-04-26）

**何が起きたか:**
HMM Day 3 で複数 K と特徴量を試した結果、 状態数を減らす(K=2)とARI=1.000で完璧に安定する代わりに、DOWN/UP 分離が消える（DOWN平均がプラスになる等）。状態数を増やす(K=4/5)と分離度は出るが ARI が 0.4 台に落ちる。

**Why:**
BTC 5年データに「明確な2状態構造」しか存在しない可能性。K=3 以上を要求するとHMMが「どう分割するか」で迷い、局所最適解が分散する。

**How to apply:**
- ARI ≥ 0.9 達成不可能でも、「合格 seed 内 ARI = 1.0」が確認できれば実質凍結可能（hmm_1h_K3_frozen.pkl の K=3・seed=13 採択がこのパターン）
- 全体 ARI が低いだけで凍結禁止と判断しない。 **合格解の収束 seed 数と内部一致度** を見る
- K=2 は候補B合格しないので採用不可

### L-134: HMMで異種特徴(ボラ・Funding等)を方向系と混ぜると支配される（2026-04-26）

**何が起きたか:**
- 方向系3本(ma20_dev/ma50_slope/di_diff)に **ボラ系2本(atr_14_pct/bb_width_20)** を追加 → ARI=0.92 だが候補B合格 0/20（HMMがボラ大/中/小で分離）
- 同3本に **Funding Rate生値** 追加 → 合格 2/20 に減少（HMMがFunding軸支配）
- 同3本に **Funding加工特徴2本(24h_sum/zscore_30d)** 追加 → 合格 0/20

**Why:**
StandardScaler後でも HMM は「分散が大きく綺麗に分離する次元」を優先選択する。価格特徴(±数%)と Funding(±0.3%)・ボラ(±数十%)はスケールが違いすぎて、HMM のクラスタリングが新特徴の軸に支配される。

**How to apply:**
- ガウシアンHMM の特徴量追加は「**方向系のみ**」が原則
- ボラ/出来高/Funding/L-S Ratio 等の異種特徴を入れたいなら HMM特徴量に直接入れず、**HMM出力の後処理** か **2階層モデル**（方向HMM + 異種HMM の出力統合）にする
- 「特徴量を増やせば精度上がる」という ML の常識は HMM では成り立たない

### L-132: HMM特徴量設計は「拡張前に冗長確認」（2026-04-26）

**何が起きたか:**
HMM Day 2 で 8 特徴量の骨組みを実装したところ、A方向性カテゴリの 4 特徴（ret_14d / ma50_dev / di_diff / dist_high_30d）すべてが互いに |r| 0.77〜0.86 の高相関で冗長と判明。当初予定の「15-20本まで拡張」を途中で止めて、まず冗長削除を優先した。

**Why:**
冗長特徴を抱えたまま拡張すると、HMMの次元数だけ膨らむが情報量は増えず、方向性カテゴリの過剰代表で状態判定が支配されるリスク。事前に削減せず20本まで膨らませて後からPCAで削るより、初期段階で相関ヒートマップ視認 → 冗長削除する方が早い。

**結果:**
- 削除: ret_14d / di_diff / dist_high_30d
- 残し: ma50_dev（方向性代表）
- 追加: hhll_30d → streak_max_30d（さらに ma50_dev と +0.82 相関だったため差し替え）
- 最終 8 特徴で |r|≥0.7 ペアゼロ達成

**How to apply:**
新規特徴量設計時は、最初の数本で必ず相関ヒートマップを確認してから拡張本数を決める。「カテゴリ代表 1 本」の原則を守り、同一カテゴリ複数本を同時投入しない。冗長は HMM だけでなく一般の機械学習でも次元の呪いの起点になる。

### L-130: 判定基準の閾値はデータ駆動で決める（根拠なき数字を提示しない）（2026-04-26）

**何が起きたか:**
HMM 状態評価基準として「UPTREND平均日次リターン > +0.20%」を提示。ユーザーから「根拠は？」と指摘されて確認したところ、BTC 5年分の実データでは平均 +0.200% / σ 3.405% で、提示した±0.20%は0.06σ相当（無意味な閾値）だった。

**Why:**
基準閾値を出す際、対象データの統計（平均・標準偏差・パーセンタイル）を確認していなかった。「±0.20%なら有意に正/負」と直感で置いただけ。BTC日次の47%が±0.20%超えており、ノイズに埋もれる閾値だった。

**How to apply:**
- 閾値や基準を出す前に、対象データの統計（mean / std / percentiles）を必ず先に算出
- 「平均±N×σ」のような正規化された数字で提示する
- BTC日次リターン分布: mean=+0.20% / σ=3.40% / [-39.5%, +19.5%]（5年分・1826日）
- 採用基準: 平均±0.3σ → UP > +1.22% / DT < -0.82% / RG ±0.34%以内（候補B）

### L-129: regime判定改善は手動ルール+標準ML共に30-40%が天井（2026-04-26）

**何が起きたか:**
ground truth（まこさん肉眼判定365日）に対して 25 個の手動ルール + 機械学習（RF/GB・特徴量50個）を試行。すべて 30-40% 正解率が天井で、全曜日80% 達成は不可能と確定。

**Why:**
1. ground truth が「週終値>週始値」基準 → 月曜時点判定 = 5日後の終値予測 = 未来予測タスク
2. ラベルに「押し目/戻り売り/スパイク/天井打ち/トレンド転換」等の繊細な区分があり、機械の特徴量（日足MA・ADX等）では追従不能
3. 365日×50特徴では機械学習に圧倒的にデータ不足（通常数千日必要）
4. 学習正解率98-100% / 検証正解率31-32% = 重度の過学習

**How to apply:**
- 教師あり学習で ground truth に完全フィットさせるアプローチは断念
- 教師なし学習（HMM）+ リターン分布評価へ路線転換
- ground truth を「正解として強制せず」、「視覚確認のリファレンス」として使う
- 5年分以上のデータ確保 + 状態を経済的有意性で評価する設計が筋

### L-128: replay_csv.py の regime_map 系3関数すべてに look-ahead バグ（2026-04-25）

**何が起きたか:**
`runner/replay_csv.py` の `_build_regime_map`（日足）/ `_build_regime_map_v3` / `_build_regime_map_hourly` 3関数すべてに同一構造の look-ahead バグを確認。`resample("D")` で「その日の23:55 close」を含む日足を作成→その日の指標計算→その日の trade 全部に同じ regime 適用 = **最大23時間55分の look-ahead**。

**実害（365d Replay 比較）:**
| 指標 | LOOKAHEAD（現行） | SAFE（shift(1)） | 差 |
|---|---|---|---|
| total NET | $5593.30 | $4506.40 | **-$1086.90 (-19.4%)** |
| P23 $/dt-day | $21.32 | $16.34 | **-$4.98 (-23.4%)** |
| P21 $/dt-day | $11.41 | $10.20 | -$1.21 |
| P2 $/dt-day | $4.75 | $5.11 | +$0.36 |

**Why:**
- 日足 close.rolling().mean() 自体は backward-looking で問題ない
- 問題は「t日のregime値が、t日のclose（=23:55値）を含めて計算」されている点
- ADX_14 は Wilder smoothing で当日値が 1/14 混入 → 境界日（ADX≒20、close≒MA70）の振り分けが変わる
- DT日数は 143→144日（+1日）でほぼ不変だが、境界日に P23 SHORT の有利な発火がプレミアムとして乗っていた

**How to apply:**
- 修正版: `--regime-safe` フラグ実装済み・3関数すべてに `lookahead_safe` パラメータ追加
- 修正は `close.shift(1)` で「t日 = t-1日までの close/high/low から計算」
- 既採用パラメータ（P21/P2/P23）は look-ahead プレミアム前提で最適化されたもの → regime 判定確定後に再評価必要
- ただし regime 判定の根本見直し（HMM）が先 → look-ahead 修正は後回し（L-129 path）

### L-126: regime 判定の粒度誤認 — 月単位が本丸（2026-04-25）

**何が起きたか:**
ユーザーが画面で「DT/UPが逆」「RGが大変動の月になっている」と肉眼で指摘。3ラウンド（hourly+ヒス36h / BB両合意 / スコアリング型v3）で日次/時間粒度のチューニングを試行したがすべて実NET悪化（-$2007/-$537）。Claudeは「regime判定改善 ≠ 実NET改善・Priorityシグナル品質が本丸」と誤結論し廃案を提案。ユーザーから「1か月単位で正しい判定ができていないと5分足シグナル最適化の威力が発揮できない」「諦めるな」と指摘されセッション切替。

**Why:**
1. 5分足 trade の avgHold = 60〜130分 → 月単位の支配 regime で十分（trade 内で regime 切替不要）
2. 日次/時間粒度でフリップを抑える方向は本質ではない。月単位の人間の肉眼判定と一致することが正解
3. 「mismatches=0」のような机上指標は実NET改善と相関しない。月内日次の品質まで含めて初めて意味がある
4. v3 で UP判定が増えて P1 損失増えたのは、UP 期間内の日次に下げ局面が混入していたから。これを「P1 の構造的欠陥」と早計判断したのが Claude のミス

**How to apply:**
- regime 判定の評価軸は「**月単位の支配 regime が人間の肉眼判定と一致するか**」のみ
- 日次切替回数や mismatches などの机上指標は二次的
- 5分足 trade の世界観で「日内regime切替」は考えなくていい（avgHold短いため）
- regime判定改善は最低10回以上の調整を辞さない（feedback_regime_visual_check）
- 本丸は「月初〜月末の支配 regime をどう確定するか」の設計。v3/hourly コードは残置済み・廃案ではなく次設計の参考素材

### L-124: BB middle slope 単独活用は構造的ノイズ過多（2026-04-25）

**何が起きたか:**
3ロジック試行の2つ目。BB middle slope (5日窓) と MA70 slope (5日窓) の両合意で trend 判定するロジック①、MA70 主導で BB が明確逆方向のみ MIXED 認定するロジック② を試行。

- BB middle slope を 5日 → 14日 など長期化する案は未試行（残課題）

### L-122: P4 単独・パラメータ最適化のみで RANGE 戦略を組むのは構造的に不可能（2026-04-25）
