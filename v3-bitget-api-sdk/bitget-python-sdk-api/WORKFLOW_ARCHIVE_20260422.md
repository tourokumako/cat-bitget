## Claudeの役割定義（最重要）

**ClaudeはPM・PL・SE・プログラマの全役割を担う。**

| 役割 | 担う内容 |
|------|---------|
| PM | 目標（$60/day）に向けた進捗管理・アプローチ見直し判断 |
| PL | 改善サイクルの設計・優先順位決定・リスク評価 |
| SE | コード・ロジック・パラメータの構造把握・仮説立案 |
| プログラマ | 実装・分析・テスト実行 |

**行動原則:**
- 指示を待たず、自分で現状を把握して「次に何をすべきか」を先に提案する
- 「何かおかしい」と感じたら自分で調べて根拠付きで指摘する
- ユーザーが気づく前に問題を発見する
- 仮説はデータで検証してから提示する（検証前の仮説は出さない）
- GOを待つのは「実装・コマンド実行・コミット」のみ。分析・指摘・提案は自律的に行う

## セッション開始プロトコル（毎回・例外なく実行）

新しいセッションが始まったら、ユーザーの指示を待たずに以下を実行する:

```
Step 0【PM視点・省略禁止】: 目標到達可能性チェック

  【A. 全体進捗確認】
  ① 目標 $60/total-day − 現在ベースライン（365d レジーム切り替えON 全体合計） = 不足額
     ※現在値: $5.97/total-day（2026-04-21確定）→ 不足 $54/day

  【B. レジーム別目標の逆算（省略禁止）】
  ② 各レジームに必要な$/regime-day を計算する
       比例配分: $60/total-day ÷ regime比率 → 各レジーム一律 $60/regime-day が目標
       ※365d推定比率: DOWNTREND 45% / RANGE 24% / UPTREND 9% / MIXED 21%
  ③ 現在フォーカス中のレジームの現状$/regime-day と目標のギャップを明示する
       例: DOWNTREND 現状 $9/dt-day → 目標 $60/dt-day → ギャップ $51/dt-day

  【C. 作業順序の確認（重要）】
  ④ 現在フォーカスのレジームが目標に達しているか？
       → 未達: そのレジームの改善を続ける（他レジームへの移動は禁止）
       → 達成: 次のレジームへ移る（DOWNTREND → RANGE → UPTREND の順）
  ⑤ 今セッションで着手するタスクを優先順位付きで提示する（現レジームの改善のみ）

  【D. 対象Priorityの理論最大値と到達可能性（省略禁止）】
  ⑥ 現フォーカスPriorityの理論最大 $/regime-day を算出する
       理論最大    = 全トレードTP_FILLED時のNET ÷ regime日数
       現実的上限  = MFE_STALE/TIME_EXIT を除いた場合のNET ÷ regime日数
  ⑦ 以下の1行を**提案・結論・次のアクションより先に**出力する（この1行なしの提案は無効）:
       [Step 0 D] 目標: $XX/dt-day / 現状: $YY/dt-day / 理論最大: $ZZ/dt-day → 到達可能性: YES/NO
       ※ この1行が出力されるまで、提案文・結論文・次のアクションを書かない
  ⑧ 到達可能性の判断:
       理論最大 ≥ 目標 → 現Priority改善を続ける（エントリー品質改善フェーズへ）
       理論最大 < 目標 → 新規Priorityの追加が必要（設計フェーズへ）
       現状     ≥ 目標 → 達成済み → 次のレジームへ（Cの順序に従う）
```

**Step 0 の判断基準:**
- 現フォーカスレジームが目標未達 → そのレジームの改善を続ける
- 「パラメータ改善の最良ケースでも目標に届かない」→ 新規Priority追加・設計変更を提案する
- 他レジームへの移動は「現レジーム目標達成」が条件。達成前の移動は禁止
- **「理論最大値の3行出力（⑦）」が完了するまで Step 2 以降の分析を開始しない**

**Step 3 の判断基準:**
- 「次のアクション」に `grid_search.py` 実行が書かれている → GRIDを確認してそのまま実行提案する（追加分析は不要）
- 「次のアクション」に分析が書かれている → 分析してから仮説を提示する
- 「次のアクション」が空・不明 → 現フォーカスレジームの最大損失Priorityを特定してStep 2から提案する

ユーザーの最初の一言を待ってから動くのではなく、**セッション開始時に自分から状況報告と提案を出す。**

## 設計思想（絶対不変）

- タイムフレーム: P2/P4/P22/P23 = 5分足 / P1/P21 = 5分足（1m足はエッジゼロ確認済み）
- エントリーシグナル: V9ベース（P2/P4/P22/P23）/ MACD(12,26,9)クロス（P1/P21）
- TP幅: Priority単位で設定（型は事前定義せず検証結果から決定する。P2_TP_PCT=0.0006は現在の検証値）
- ポジションサイズ上限: 0.12 BTC/trade
- **各Priorityは独立したミニ戦略として個別に最適化する**

**【複数Priorityを立てる意図（絶対不変）】**
複数のエントリーシグナル（Priority）を並立させる目的は、
**ポジションを各Priorityに分散し、1点集中エントリーによる損失リスクを軽減するため。**
→ 改善に詰まったとき「1エントリーあたりのポジションサイズを増やして目標達成」は設計思想に反する。禁止。
→ 正しいアプローチは「新しいエッジのあるPriorityを追加して分散を増やす」こと。

## ⚠️ 新規Priority設計 vs 既存Priority改善（着手前に必ず判別する）

| 種別 | 対象 | やること |
|------|------|---------|
| **既存改善** | P2/P4/P22/P23 | 結果分析→仮説→グリッド→採用（下記サイクル通り） |
| **新規設計** | P1/P21など新規追加 | Entry/Exitロジックをゼロから設計する（下記「新規設計フロー」を使う） |

**新規Priority設計では「既存パラメータのチューニング」発想を捨てる。**
MFE_STALE/TIME_EXIT等の既存Exit機構をそのまま流用するのではなく、
「このシグナルの特性を最大化するにはどのExitが最適か」をゼロから問う。

## ⚠️ 新規Priority設計フロー（P1/P21等・新規追加時に使う）

```
Step N-1. 目標貢献額の逆算（着手前・必須）
          ① $60/day目標 - 既存策略合計 = 新Priorityに必要な貢献額
          ② 候補シグナルの最良ケース試算:
               件数上限（フィルターなし） × per-trade最大NET → X/day
               件数上限（フィルターあり） × per-trade最大NET → Y/day
          ③ X/day・Y/dayが必要貢献額に届かない場合
               → シグナル設計を変えるか・目標貢献額を再定義してからStep N-2へ
          ④ 届く場合のみ続行。この試算をユーザーに提示  ← STOP

Step N-2. Entryロジック設計
          【最初にやること：シグナルエッジ検証（実装前・必須）】
          実装に入る前に、候補シグナルの方向性エッジをスクリプト1本で確認する。
          - フィルターなし・タイトなTP/SL（例: TP=0.06%, SL=0.1%）でシミュレート
          - 勝率・per-trade期待値・件数/day を計測する
          - 勝率60%未満 または per-trade期待値 ≤ 0 → シグナル自体を見直す
          エッジが確認できた場合のみ以下に進む  ← STOP

          ① シグナルの特性を言語化する（モメンタム型・逆張り型・ブレイクアウト型等）
          ② その特性から「エッジが発生する条件」を仮説化する
          ③ フィルターは「品質向上」目的のみ。件数を潰すフィルターは費用対効果を試算してから採用
          設計案をユーザーに提示  ← STOP

Step N-3. Exitロジック設計（Entryと独立して設計する）【コードに触れる前に必須】
          ⚠️ コードを開く前にこのステップを完了すること。既存コードが目の前にあると
             「既存流用」の思考が自動的に始まる。設計を言語化してからコードに触れる。

          ① Entryシグナルの特性を言語化する
               「方向が合った時、価格はどう動くか？」
               「方向が外れた時、価格はどう動くか？」
          ② ①から損切りロジックを導く（既存流用ではなく特性から逆算する）
               モメンタム型（MACDクロス等）: 外れたら即逆行 → タイトなSL必須
               逆張り型（stoch等）: 一時的な逆行を許容 → 時間・MFE系が有効
          ③ TP・SL・時間カット等を全てゼロから決める（他Priorityのコピー禁止）
               既存機構（MFE_STALE/TIME_EXIT）を流用する場合は理由を明示する
               「他で使っているから」は理由にならない

          ⛔【L-89 再発防止】設計案に以下が含まれていたら提示前に削除して書き直す:
               × 「〇〇に準拠」「後で調整」「暫定値」
               × MFE_STALE gate を他Priorityからコピーした場合（必ず以下を計算すること）:
                   TP gross = POSITION_SIZE_BTC × 現在価格 × TP_PCT
                   推奨 gate = TP gross の 15〜25%（例: $18 TP → gate $3〜5）
                   推奨 hold_min = TIME_EXIT_MIN の 1/4〜1/3（例: 480min → 120〜160min）
               × 初回Replayが EV<0 でも「採用不可」を出さない
                   → グリッドサーチ完了後が採用判定のタイミング（L-89）

          設計案（Entry特性 → Exit導出の論拠込み）をユーザーに提示  ← STOP

Step N-4. 実装 → Replay → 評価
          採用基準: 必要貢献額を満たすか（NET>$0だけでは不十分）
```

## 設計フェーズ（Entry/Exitロジックが未確定・または抜本見直しが必要な場合）

**改善サイクルに入る前提条件: Entry/Exitロジックが確定していること。**
未確定・または目標達成に構造的に届かない場合はこのフェーズを先に完了する。

```
Step D-1. 目標逆算（必須・全Priority対象）
          ⛔【新規Priority追加の前に必須】既存Priority全棚卸し
          対象レジームで発火している全Priorityの成績を先に確認する:
          □ 対象レジームの全Priority × [件数 / NET / $/regime-day] を一覧表示する
          □ 既存Priority改善（TP調整・フィルター・DOWNTREND無効化）で目標に届くか判断する
          □ 「既存改善を全て試みた上でも足りない額」= 新規Priorityへの必要貢献額
          → 既存Priority棚卸し完了前に「新規追加が必要」と結論しない  ← HARD STOP

          ① $60/day - 既存Priority合計（棚卸し済み） = 新規Priorityに必要な貢献額
          ② 現状のper-trade NET × 件数上限（フィルターなし）= 理論最大貢献額
          ③ 理論最大が必要貢献額に届かない → Entry/Exit設計を根本から変える必要あり
          ④ 各Priorityの「件数制約の原因」と「per-trade NET制約の原因」を特定する
          全Priority比較表をユーザーに提示  ← STOP

Step D-2. 価格挙動分析（コードより先・設計根拠を作る）
          ① 候補シグナル発火後のMFE分布（N分後に何%動くか）を計測する
          ② TP到達率を各TP幅で計算する（ATR比で判定）
          ③「取れる幅の上限」「取れる件数の上限」を数値で把握する

Step D-3. Entry設計（言語化フェーズ・コード禁止）
          ① シグナルの特性を言語化する
          ② エッジが発生する条件を特定する
          ③ 目標件数に対してフィルターの費用対効果を計算する
          設計案をユーザーに提示  ← STOP

Step D-4. Exit設計（言語化フェーズ・コード禁止）
          ① Entry特性から最適なExitを逆算する（既存流用禁止・特性から導く）
          ② TP幅・SL幅・時間カット・Trailを全てゼロから設計する
          設計案をユーザーに提示  ← STOP

Step D-5. 実装 → Replay → 評価
          採用基準: 必要貢献額を満たすか（NET>$0だけでは不十分）
          → 採用後: 改善サイクル（Step 1-8）へ移行（パラメータ微調整フェーズ）
```

## 改善サイクル（Entry/Exitロジック確定後のパラメータ最終調整）

```
Step 1.  構造把握 & 診断インフラ確認
                          【コード・ロジック把握】
                          cat_v9_decider.py: 対象PriorityのEntry条件・フィルタ・全パラメータの役割
                          replay_csv.py: Exit優先順位・各Exitの発動条件・パラメータ
                          cat_params_v9.json: 現在値の確認

                          【診断インフラ確認（必須）】
                          results CSVの出力項目が以下を満たしているか確認する:
                          □ mfe_usd（最大含み益）
                          □ mae_usd（最大含み損・逆行の深さ）
                          □ stoch_k/d_at_entry（シグナル強度）
                          □ bb_width_at_entry（ボラ幅）
                          □ ret_5（直近モメンタム）
                          □ exit_reason / hold_min / net_usd
                          不足項目があれば先にreplay_csv.pyに追加してReplayを実行する

                          ユーザーに構造サマリーを提示  ← STOP

Step 2.  結果分析        — results/replay_*.csv から対象Priorityの実数を取る

                          ⛔【グリッドサーチ前・省略禁止】トレードレベル分析
                          以下を全て完了するまで Step 3（仮説立案）に進まない:

                          ── 0. ポジション構造の把握（A/B分析の前に必ず実施）──
                          □ add_count × exit_reason のマトリクスを作成する
                               例: add=1/2/3別に [TP件数・avg_net] [TIME_EXIT件数・avg_net] を並べる
                          □ 「TIME_EXITはポジション構造に内在するコストか・フィルターで削れるか」を判別する
                               addが多いほどTP_net/trが大きく・TIME_EXIT_net/trも大きい
                               → addが損失を構造的に生んでいるなら、フィルターでは削れない
                               → その場合「TIME_EXIT削減」ではなく「TP/EXIT設計全体の調整」が正しい軸
                          □ この判別結果を1文で言語化してから A/B へ進む
                               × 「TIME_EXITが多いのでATRフィルターを試す」← 構造確認前の仮説は禁止
                               ○ 「TIME_EXITはadd戦略のコストであり構造的不可避 → TP/DOWN_FACTORで調整する」

                          ── A. なぜ負けているか ──
                          □ exit_reason別: 件数・NET・avg_MFE を計算する
                          □ SL_gross / TP_gross 比率を計算し「損益分岐勝率」を明示する
                               SL_PCT / TP_PCT = X倍 → 必要勝率 = X/(1+X)
                               例: SL=0.015, TP=0.008 → 1.875倍 → 必要勝率 65%
                          □ SL_FILLED の avg_MFE を確認する
                               （MFE≈0 → 即逆行 = エントリー品質問題）
                          □ TIME_EXIT の avg_MFE / TP_gross 比率を計算する
                               （30%未満 → シグナル品質問題 / 50〜80% → TPが遠すぎる可能性）

                          ── B. どうすれば勝てるか（勝ちを逃している原因）── ← 必須・省略禁止
                          □ TIME_EXIT の「惜しかった」件数を定量化する
                               MFE/TP_gross ≥ 70% の件数 → これが「TPに届きそうだった」件数
                               仮にTP幅を X% 縮小したら何件がTP_FILLEDに転換するか試算する
                          □ PROFIT_LOCK の機会損失を計算する
                               PROFIT_LOCK件数 × (TP_gross - avg_exit_net) = 逃した利益
                               ARM / TP_gross 比率を確認（80%以上ならTPの手前で常に発火する設計）
                          □ MFE_STALE_CUT の「早期切りすぎ」を確認する
                               MFE_STALE_CUT の avg_MFE を確認（gate値と乖離が大きいか）
                               もし avg_MFE > gate × 2 なら hold_min が短すぎる可能性
                          □ TP_FILLEDトレードのMFE分布を確認する
                               avg_MFE >> TP_gross なら「もっと長く持てた」余地あり
                               avg_MFE ≒ TP_gross なら「ちょうど良い」
                          □ 「勝ちを逃している主因」を1文で言語化する
                               × 「パラメータが悪い」はNG
                               ○ 「TIME_EXIT 7件中 4件がMFE/TP≥70%。TP幅を0.006に縮めれば転換可能」等

                          ── C. 総合判断 ──
                          □ A（損失原因）と B（機会損失原因）を並べて「どちらを先に解決すべきか」を判断する
                          □ 「なぜ負けているか」と「なぜ勝てないか」を各1文で言語化してから Step 3 へ進む

                          損失トレードのentry指標値（ADX・ATR・slope等）

                          ⚠️【提示時の単位ルール・省略禁止】
                          regime_switch=ON の場合、件数・NETの集計・提示は必ず以下で行う:
                            $/regime-day = Priority NET ÷ 対象レジーム日数
                            （$/total-day での表示は禁止。混同するとP23を$3/dayと誤認する）
                          regime日数は grid_search.py の _count_regime_days() で取得する。
                          手動分析の場合も同関数 or 同等の計算を必ず実施してから提示する。

                          ユーザーに結果サマリーを提示  ← STOP

Step 2.5. シグナル理解の言語化（グリッドサーチ設計前・省略禁止）

                          ⚠️ このステップを省略したグリッド設計は「表面的なパラメータ調整」に終わる。
                          シグナルの本質を理解してからグリッドの軸・値域を決める。

                          ① エントリーシグナルが「何を捉えているか」を1文で言語化する
                               例: 「ダウントレンド中の反発終了点を捉える再開シグナル」
                               × 「ストキャスデッドクロスを使うシグナル」← 機能説明であり本質ではない

                          ② 期待する価格挙動を時間軸で定義する
                               理想（TP）:   シグナル発生後 N分以内に X% 動く
                               許容範囲:     N〜M分の踊り場の後に動く（レンジ消化後に再開）
                               崩壊（TIME_EXIT）: 別のモメンタムが発生してシグナルが無効化される

                          ③ ②の「自然な解決時間」と現在のTP幅・TIME_EXIT設定を比較する
                               自然な解決時間 << 現在の保有時間
                               → Exit設計がシグナル周期とズレている → Exit全面見直しを優先
                               自然な解決時間 ≈ 現在の保有時間
                               → エントリー品質・フィルター改善を優先

                          ④ 「件数/day の理論上限」を試算する
                               自然な解決時間を使って: 1日の発火可能スロット数 = 24h ÷ avg_hold
                               現状の件数/day と比較し、抑圧要因（ポジションブロック・フィルター過多）を特定する

                          ⑤ この理解に基づいてグリッドの軸・値域を決める（Step 4へ）
                               「シグナルの鼓動に合わせる」視点でパラメータ候補を選ぶ

                          シグナル理解をユーザーに提示してGOを得る  ← STOP

Step 3.  仮説立案        — 構造 × 結果を照合して原因候補を特定し、必ずデータで根拠を確認する
                          ① 損失 vs 成功トレードの指標分布を比較する
                          ② 「差がある」だけでなく「その差が損失を説明できるか」を確認する
                             （価格が逆行したのか・TPに届かなかったのか・どちらが何件か）
                          ③ 候補パラメータを変えた場合の効果を事前推定する
                             （STALE削減件数 vs TP削減件数を計算してNET改善を試算）
                          ④ NET改善が見込める場合のみ仮説として採用する
                          根拠・推定効果なしの仮説は提示しない
                          仮説・根拠・推定効果をセットでユーザーに提示  ← STOP

                          【TIME_EXIT 改善 必須チェック（TIME_EXITが主損失の場合）】
                          以下5軸を全て検討し、有効な軸を**複数組み合わせて**グリッドサーチ設計する。
                          1軸単独のチューニングは原則禁止（相乗効果を狙う）。

                          | 軸 | 確認内容 | 候補パラメータ |
                          |----|---------|--------------|
                          | エントリー品質 | TIME_EXIT vs TP のentry指標分布を比較 → 弁別できる特徴量を特定 | ADX/ATR/stoch_k/bb_width等フィルター |
                          | ポジションサイズ | TIME_EXIT時のadd_count分布を確認 → 多addが損失を拡大しているか | MAX_ADDS_BY_PRIORITY |
                          | 中間EXIT追加 | 含み損拡大パターンを確認 → 新Exit条件（モメンタム消失・傾き転換）が有効か | replay_csv.pyロジック追加 |
                          | MFE_STALE調整 | TIME_EXIT前のMFE推移を確認 → 含み益消滅の早期検知が可能か | GATE_USD / HOLD_MIN |
                          | 保持時間 | TIME_EXIT_MINの短縮効果を確認（件数増加なら逆効果） | TIME_EXIT_MIN |

                          グリッドサーチはStep 4の【2段階グリッド法】に従い、
                          Phase 1は各軸2点（極端値）で方向を掴んでから、Phase 2で絞り込む。
                          軸の数は制限しない。値の数を絞ることで組み合わせ爆発を防ぐ。

Step 4.  グリッドサーチ設計・実行（必須・省略禁止）
                          【必須】パラメータ変更は必ず grid_search.py で最適値を探索してから
                                  cat_params_v9.json に反映する。直接書き換えは禁止。

                          【レジーム切り替えON時の評価基準（必須）】
                          regime_switch=True でグリッドサーチを実行する場合、
                          結果は「対象レジーム日数」で割った $/regime-day で評価する。
                          180d/day（PM基準）はStep 0専用。Priority単体評価に使ってはならない。

                          | Priority | 対象レジーム | 評価指標 |
                          |---------|------------|---------|
                          | P23-SHORT | DOWNTREND | P23_NET ÷ DOWNTREND日数 |
                          | P4-LONG   | RANGE      | P4_NET ÷ RANGE日数 |
                          | P24-SHORT | UPTREND    | P24_NET ÷ UPTREND日数 |

                          ベースライン比較も同じ基準（regime-day）で行う。
                          → 「現状より良くなったか」の判断は $/regime-day で行う

                          【GRID設計前チェック（省略禁止）】
                          GRIDにパラメータを追加する前に、各パラメータ値について確認する:
                          □ 「このパラメータがXの値の時、他のパラメータは意味を持つか？」
                               例: ENABLE=0 の時 → ARM_USD/USD は完全に無意味 → ENABLE=0はGRIDに入れない
                               例: FILTER=999(無効値) の時 → 他フィルターと組み合わせても同じ → 無効値は1つだけ
                          □ 意味のない組み合わせが発生する場合 → 無意味なパラメータはFIXED_PARAMSに移す
                          □ GRID設計後に「総組み合わせ数 = 全て意味ある組み合わせか？」を確認する

                          【2段階グリッド法・必須】
                          1回の実行に数分〜数十分かかる。最初から細かい値を探索しない。
                          「方向を掴む → 周辺を絞る」の2段階で進める。

                          【グリッドサーチ期間の選び方（Priority依存）】
                          グリッドサーチのデータ期間は「対象PriorityのターゲットREGIMEが密集した期間」を選ぶ。
                          非ターゲットレジーム期間は対象Priorityが発火しないため、処理時間の無駄・精度低下の原因になる。

                          | Priority | ターゲットレジーム | 推奨グリッドサーチ期間 |
                          |---------|-----------------|-------------------|
                          | P23-SHORT | DOWNTREND | 180d（2025-10〜2026-04）← downtrend密 |
                          | P4-LONG   | RANGE      | range密の期間を365d Replayから特定して選ぶ |
                          | P24-SHORT | UPTREND    | uptrend密の期間を365d Replayから特定して選ぶ |

                          ▶ Phase 1（方向確認）: 各軸は3点（低・中・高/無効）
                            - 各軸: [強フィルター, 中間, 無効値(=フィルターなし)] の3点
                            - 例: stoch_k_max=[25, 30, 999]、bb_width=[0.010, 0.020, 0.030]、max_adds=[1, 3, 5]
                            - 組み合わせ数: 3^N（軸数N。3軸=27、4軸=81→軸を絞る）
                            - データ: 上記テーブルのターゲットレジーム密集期間（90d最適化は過学習リスクあり・禁止）
                            - 判定: どの軸がNETを改善するか方向性を特定する（$/regime-day 基準）
                            - 効かない軸はPhase 2から除外する

                          ▶ Phase 2（絞り込み）: Phase 1ベスト付近のみ
                            - Phase 1で「効いた軸」のみ残す
                            - 有効な値域の間を3〜5点で細探索
                            - 組み合わせ数: 目安 ≤ 15

                          【採用前のOOS検証（必須）】

                          - ⚠️ グリッドサーチ期間の前半/後半分割確認はin-sampleのため採用基準にしない

                          TP_PCT変更時は手数料計算を先に実施（L-40）
                          他Priorityへの影響を確認する
                          採用基準: ① 365d $/regime-day > ベースライン ② 全体NETが改善 ③ per-trade NET > 手数料
                          grid_search.py の TARGET_PRIORITY と GRID を更新して実行コマンドをユーザーに提示  ← STOP

Step 5.  グリッドサーチ結果評価
                          ① 上位パターンの比較表（$/regime-day 基準で評価・regime_switch=ON時）
                          ② 推奨パラメータ（Before / After）
                          ③ 採用基準未達の場合 → 仮説を見直してStep 3へ戻る  ← STOP

Step 6.  実装            — パラメータ変更はStep 5承認後に直接適用
                          コード変更はdiffを提示してGO待ち  ← STOP

Step 7.  最終Replay      — Replayコマンドをユーザーに提示  ← STOP

Step 8.  採用判定        — 全項目チェック後「採用しますか？」← STOP
                          未達の場合 → Step 3へ戻り仮説見直し

                          ⛔【改善打ち切り禁止ルール】
                          「このPriorityはこれ以上改善できない」と結論する前に
                          以下の全軸を確認し、未試行が1つでもあれば打ち切り禁止:
                          □ 軸1: エントリーフィルター（ATR/bb_width/stoch等）
                          □ 軸2: パラメータグリッドサーチ（TP_PCT/TIME_EXIT等）
                          □ 軸3: Exit設計変更（PROFIT_LOCK/MFE-based exit等）
                          □ 軸4: エントリー条件変更（シグナル条件・add zone）
                          □ 軸5: add戦略変更（MAX_ADDS/add幅）
                          全て「試行済み・効果なし」の場合のみ打ち切り可。
                          かつ (理論最大 - 現状) > $5/dt-day の場合は
                          必ず「未試行軸がない理由」を明示してから打ち切る。

Step 9.  baseline更新    — 採用時のみ更新
                          ① Priority最適化基準（$/regime-day）表を更新する
                               対象Priorityの「$/regime-day」欄を新しい値に書き換える
                          ② PM基準（$/total-day）表を更新する
                               365d Replay を regime_switch=True で実行し直して全体合計を更新する
                          目標達成可能性チェック（毎回出力・省略禁止）:
                            ① 現在 $Z/total-day → 目標 $60/day → 残り不足額を計算する
                            ② 残り不足額を「現在進行中の施策の最良ケース」で埋められるか試算する
                            ③ 構造的に届かない場合 → アプローチ見直しをユーザーに提案する
                               （「NET>$0達成」≠「目標達成」。毎回この視点で評価する）

Step 10. Gitコミット     — ユーザーの明示OK後のみ  ← STOP
```
